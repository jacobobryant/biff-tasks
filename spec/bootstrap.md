# Bootstrap spec

## Commands covered

- `setup`

## Purpose

`setup` prepares a freshly cloned Biff app repo and remains safe to rerun later
as the default generated files evolve.

## Behavioral contract

- `setup` MUST remain idempotent enough to rerun on an existing repo without
  destroying user work.
- On first run, `setup` MUST rename the template main namespace away from
  `com.example`.
- `setup` MUST generate any missing local config files from templates.
- `setup` MUST ensure Tailwind is installed when the project uses the local-bin
  Tailwind path.
- `setup` SHOULD install a pre-commit hook that:
  - runs `clj -M:run format`
  - fails if that command reformats tracked files
  - does not run the test suite
- `setup` SHOULD manage an autogenerated GitHub Actions workflow that runs the
  `test` task at `.github/workflows/tests.yml`.
- The autogenerated workflow SHOULD be overwritten on rerun and SHOULD include a
  clear "autogenerated, do not edit" comment.

## Generated artifacts

The initial default generated artifacts are:

- local config files (`config.env`, `config.prod.env`)
- pre-commit hook(s)
- a GitHub Actions test workflow

## Recommended initial direction

- Start with only a test workflow, not a deploy workflow.

## Non-goals

- Automatic production deployment from GitHub Actions is out of scope for the
  default generated workflow.
- `setup` is not responsible for preserving compatibility with old Biff task
  names or config keys.

## Open questions

- Provisioning is intentionally not owned by `setup`; it will be specified
  separately in production-task specs.
